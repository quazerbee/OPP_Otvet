# Методичка к экзамену: все 30 вопросов

**Как устроен каждый вопрос:**
1. 🎓 **Чёткий ответ** — то, что говоришь у доски (правильные термины).
2. 🔑 **Сложные слова** — расшифровка каждого термина.
3. 🔬 **Пример живьём** — конкретно, по шагам.
4. 💡 **По-человечески** — короткий итог.

## Опорные грамматики (используются во всём документе)

Чтобы примеры были сквозными, договоримся о нескольких маленьких грамматиках:

- **Г1:** `S → aS | b` — делает `b, ab, aab, …` (сколько-то `a`, потом `b`).
- **Г2:** `S → AB`, `A → a`, `B → b` — делает только `ab`.
- **Г-скобки:** `S → (S) | x` — делает `x, (x), ((x)), …`.
- **Г-пустая:** `S → aS | e` — делает `e, a, aa, …` (`e` = пусто).
- **Арифметика:** выражения вида `a + a * a` — только где нужны приоритеты.

# 1. Проблематика разработки языков программирования

🎓 **Чёткий ответ.** Разработка ЯП — многоэтапный процесс: (1) анализ предметной области и сбор функциональных требований; (2) определение алфавита и лексики, «ручное» моделирование текстов программ; (3) формализация — описание языка в БНФ, синтаксических диаграммах, грамматиках; (4) создание реализуемого стандарта. Ключевая связанная задача — разделение **синтаксиса** (структуры правильных предложений) и **семантики** (корректности смысла).

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| предметная область | для чего язык, какие операции и данные нужны |
| синтаксис | правила «как из символов строить правильные конструкции» |
| семантика | правила «имеет ли конструкция смысл» |
| БНФ | формальная запись правил языка (нотация) |

🔬 **Пример живьём.** Синтаксис против семантики. Фраза «автомобиль едет» и «автомобиль думает» **обе синтаксически правильны** (подлежащее + сказуемое), но вторая бессмысленна — это семантика. В коде: `sum := a + b` синтаксически верно всегда, но если `a = 10.5`, а `b = истина` — **семантически неверно** (нельзя складывать число и логическое значение).

💡 **По-человечески.** Синтаксис описать легко (есть стройная теория), а семантика — самая трудная часть; её часто задают неформальными «соглашениями».

# 2. Проблематика реализации языков программирования

🎓 **Чёткий ответ.** Реализация ЯП — это сам транслятор плюс платформа (железо + ПО), а в широком смысле — весь процесс создания транслятора от проекта до отладки. Перед разработчиком встают три задачи: на каком языке писать компилятор; как получить его машинный код; как перенести на другую платформу. Ответом на них стали технологии **«раскрутка»** и **«компиляция компилятора»**.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| реализация | транслятор + платформа, на которой он работает |
| платформа | конкретное железо и операционная система |
| перенос | заставить компилятор работать на другой машине |

🔬 **Пример живьём.** Насколько реализация сложна: между появлением языка **Ада** и созданием первого компилятора для него прошло **более 10 лет**.

💡 **По-человечески.** «Реализовать язык» — это не придумать синтаксис, а построить работающий транслятор и заставить его жить на нужном железе.

# 3. Транслятор, компилятор, фазы компиляции

🎓 **Чёткий ответ.** **Транслятор** переводит исходную программу в эквивалентную объектную. **Компилятор** — транслятор с языка высокого уровня в объектный (машинный) код. **Интерпретатор** — транслятор, который сам выполняет программу. Фазы компиляции: лексический анализ, синтаксический анализ, семантический анализ, генерация кода, оптимизация.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| объектный код | результат перевода (команды ассемблера/машины) |
| фаза | отдельный этап работы компилятора |
| интерпретатор | переводит и сразу выполняет, без отдельного exe |

🔬 **Пример живьём.** Один оператор `S := (a + b) * 0.17` проходит фазы:
- **лексический анализ:** нарезка на лексемы `S`, `:=`, `(`, `a`, `+`, `b`, `)`, `*`, `0.17`;
- **синтаксический анализ:** строится дерево разбора (операции `:=`, `*`, `+` — узлы);
- **генерация кода:** `LOAD =0.17 / STORE S2 / LOAD B / STORE S1 / LOAD A / ADD S1 / MPY S2 / STORE S`;
- **оптимизация:** убираем лишнее → `LOAD A / ADD B / MPY =0.17 / STORE S`.

💡 **По-человечески.** Транслятор — переводчик; компилятор — его частный случай (в машинный код); интерпретатор — переводит и тут же исполняет. Фазы идут конвейером.

# 4. Инструменты и технологии разработки и реализации ЯП

🎓 **Чёткий ответ.** Инструменты образуют иерархию: среда программирования → система автоматизированной разработки ЯП → две группы инструментов. **Инструменты разработки языка:** восстановление грамматик и эквивалентные преобразования грамматик. **Инструменты разработки компилятора:** генераторы сканеров, генераторы синтаксических анализаторов, автоматические генераторы кода, средства синтаксически управляемой трансляции. Базовые технологии реализации — «раскрутка» и «компиляция компилятора».

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| восстановление грамматик | по словам языка получить порождающую грамматику |
| эквивалентные преобразования | переделать грамматику, сохранив язык |
| генератор сканеров/анализаторов | программа, которая строит сканер/анализатор автоматически |

🔬 **Пример живьём.** Среда программирования со средствами сборки — например, утилита `make` в Unix, на которой собирают компиляторы.

💡 **По-человечески.** Это «мастерская» для создания языка: одни инструменты помогают придумать и причесать грамматику, другие — собрать по ней транслятор.

# 5. Технология раскрутки

🎓 **Чёткий ответ.** Раскрутка — использование самого компилируемого языка для написания компилятора с последующей трансляцией его в машинный код. Описывается **Т-диаграммами** — тройкой языков $S\,I\,T$: входной (компилируемый) $S$, язык реализации $I$, выходной (целевой) $T$. Технология итерационная: имея компилятор для маленького подмножества $L_0$, язык расширяют $L_0 \to L_1 \to \dots \to L_n = L$, на каждой стадии прогоняя новый компилятор через старый.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| Т-диаграмма | схема компилятора: вход $S$, реализация $I$, выход $T$ |
| подмножество $L_0$ | урезанная версия языка, для которой компилятор пишут вручную |

🔬 **Пример живьём.** Как учить язык по словарю **на этом же языке**: сначала дают мини-словарь из 100 базовых слов с переводом на родной (это компилятор для $L_0$). С его помощью понимаешь определения более сложных слов, потом ещё более сложных — «раскручиваешь» весь язык, всё меньше опираясь на родной.

💡 **По-человечески.** «Вытягивание себя за волосы»: маленьким готовым компилятором собираем компилятор побольше, тем — ещё больше, пока не охватим весь язык.

# 6. Математический аппарат

🎓 **Чёткий ответ.** Основа — теория множеств и связанные понятия: **отношения** (с операциями степени, транзитивного замыкания $R^+$ и рефлексивно-транзитивного $R^*$), **отображения** (функции) и **декартовы произведения** множеств. Через них строго определяют цепочки, языки, грамматики, автоматы и преобразователи.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| множество | «мешок с элементами» |
| отношение | список пар «кто с кем связан» |
| транзитивное замыкание $R^+$ | «куда можно добраться за 1+ шагов по связям» |
| отображение (функция) | таблица «вход → ровно один ответ» |
| декартово произведение | все возможные комбинации пар |

🔬 **Пример живьём.** Пусть `R = {(a,b), (b,c)}` (a связан с b, b с c). Тогда транзитивное замыкание добавляет «непрямую» связь: `R⁺ = {(a,b),(b,c),(a,c)}` — до `c` можно добраться из `a` через `b`. Ровно эта идея потом появится в выводах грамматики (`⇒*`) и в тактах автомата.

💡 **По-человечески.** Три инструмента: мешки (множества), связи (отношения), таблицы вход→ответ (функции). Из них собраны все автоматы курса.

# 7. Грамматика, классы грамматик и языков

🎓 **Чёткий ответ.** Грамматика — четвёрка $G=(N,\Sigma,P,S)$: $N$ — нетерминалы, $\Sigma$ — терминалы, $P$ — правила, $S$ — начальный символ. Классификация Хомского по виду правил: тип 0 (общего вида), тип 1 (контекстно-зависимые, КЗ), тип 2 (контекстно-свободные, КС), тип 3 (регулярные).

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| терминал | настоящий символ (`a`, `b`), остаётся в слове |
| нетерминал | «заготовка» (`S`, `A`), которую надо раскрыть |
| правило (продукция) | стрелка «что → на что заменить» |

🔬 **Пример живьём.** Грамматика **Г1** `S → aS | b` формально: $N=\{S\}$, $\Sigma=\{a,b\}$, $P=\{S{\to}aS,\ S{\to}b\}$, начальный $S$.

Классы (от вольных к строгим):

| Тип | Класс | Вид правил |
|---|---|---|
| 0 | общий | любые |
| 1 | КЗ | `α→β`, где \|α\|≤\|β\| |
| 2 | КС | `A→α` (слева один нетерминал) |
| 3 | регулярная | `A→aB` или `A→a` |

Язык `{aⁿbⁿcⁿ}` (одинаково `a`, `b`, `c`) **не является КС** — нужен тип 1.

💡 **По-человечески.** Грамматика = 4 вещи (заготовки, символы, правила, старт). Чем строже тип правил — тем проще для языка построить машинку-распознаватель.

# 8. Вывод цепочки и язык грамматики

🎓 **Чёткий ответ.** Вывод задаётся отношением непосредственной выводимости `⇒` (заменили левую часть правила на правую). Из него строят `⇒⁺` (за 1+ шагов) и `⇒*` (за 0+ шагов). **Язык грамматики** $L(G) = \{w \mid S \Rightarrow^* w,\ w$ из одних терминалов$\}$.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| `⇒` | «за один шаг превращается в» |
| терминальная цепочка | слово из одних настоящих символов |
| крона дерева | листья дерева слева направо = выведенное слово |

🔬 **Пример живьём.** Грамматика **Г2** (`S→AB, A→a, B→b`), выводим `ab`:
```
S ⇒ AB ⇒ aB ⇒ ab
```
Значит `S ⇒* ab`, и `ab ∈ L(G)`. Тут язык состоит из одного слова: `L(G) = {ab}`.

💡 **По-человечески.** Вывод — пошаговая сборка слова из старта. Язык — все слова, которые так можно собрать.

# 9. Распознаватели. КА и МП-автомат

🎓 **Чёткий ответ.** **Распознаватель** — устройство (лента + головка + управляющее устройство с конечной памятью), отвечающее «допускаю/не допускаю». **Конечный автомат (КА)** $M=(Q,\Sigma,\delta,q_0,F)$ — без памяти. **МП-автомат** — семёрка $(Q,\Sigma,\Gamma,\delta,q_0,Z_0,F)$ со стопкой (магазином). Язык распознавателя — множество допускаемых цепочек.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| состояние | «режим/настроение» машинки |
| конфигурация | снимок: настроение + остаток ленты |
| допускает | в конце оказался в «хорошем» состоянии |
| магазин | стопка (память) у МП-автомата |

🔬 **Пример живьём.**
- **КА** на слове `01001` идёт по состояниям и заканчивает в заключительном → **допущено**.
- **МП-автомат** для языка `{0ⁿ1ⁿ}` (поровну нулей и единиц): кладёт каждый `0` в стопку, на каждую `1` снимает один `0`. Стопка опустела ровно к концу → допущено. **Стопка работает как счётчик** — у КА такого нет.

💡 **По-человечески.** КА — контролёр без блокнота (только настроения). МП-автомат — контролёр со стопкой-блокнотом, поэтому умеет считать.

# 10. Регулярные выражения

🎓 **Чёткий ответ.** Регулярное выражение обозначает **регулярное множество**, строящееся из `∅`, `{e}`, `{a}` тремя операциями: объединение, конкатенация, итерация. Приоритеты: `*` → конкатенация → `+`. Применяются для компактного описания лексем.

🔑 **Сложные слова.**

| Операция | Значок | Смысл |
|---|---|---|
| объединение | `+` (или `\|`) | «или» |
| конкатенация | (подряд) | «а потом» |
| итерация | `*` | «сколько угодно раз, в т.ч. 0» |

🔬 **Пример живьём.**
- `(0+1)` = «0 или 1»;
- `(0+1)*011` = «любые нули-единицы, **заканчивающиеся на 011**»;
- `(a+b)(a+b+0+1)*` = «слова, **начинающиеся** с `a` или `b`».

💡 **По-человечески.** Шаблон-маска из трёх кубиков. Им описывают лексемы (имена, числа), а по шаблону строят КА для сканера.

# 11. Соответствия между способами описания языков

🎓 **Чёткий ответ.** Каждому классу грамматик Хомского соответствует класс распознавателей, определяющий тот же класс языков. Дополнительно на уровне регулярных языков взаимно соответствуют регулярная грамматика, синтаксическая диаграмма, регулярное выражение и НКА. Важнейшее для практики — КС-грамматика ↔ МП-автомат: по КС-грамматике строят МП-автоматы $R_l$ и $R_r$, моделирующие левые и правые выводы.

🔑 **Сложные слова.** $R_l$/$R_r$ — два МП-автомата для КС-грамматики: один моделирует левый вывод (для нисходящего разбора), другой — правый (для восходящего).

🔬 **Пример живьём.**

| Грамматика (порождает) | ↔ | Распознаватель (проверяет) |
|---|:---:|---|
| праволинейная (тип 3) | ↔ | конечный автомат |
| КС (тип 2) | ↔ | МП-автомат |
| КЗ (тип 1) | ↔ | линейно-ограниченный автомат |
| общая (тип 0) | ↔ | машина Тьюринга |

💡 **По-человечески.** Грамматика и автомат — одно и то же с двух сторон: одна слова собирает, другой проверяет. Чем сложнее язык — тем мощнее нужен автомат.

---

# 12. Расширенные регулярные выражения (РРВ)

🎓 **Чёткий ответ.** РРВ — это обычные регулярные выражения, дополненные новыми операциями: позитивная итерация `R⁺`, ограниченные итерации `R*ⁿ`, `R⁺ⁿ`, разность `R₁−R₂`, пересечение `R₁∩R₂`, а также регулярные определения (имена). Цель — более компактное описание лексем.

🔑 **Сложные слова.**

| Операция | Смысл |
|---|---|
| `R⁺` | повтор **1 и больше** раз (а `*` — от нуля) |
| `R₁−R₂` | подходит под `R₁`, но не под `R₂` |
| `R₁∩R₂` | подходит под оба сразу |
| регулярное определение | имя для шаблона (как переменная) |

🔬 **Пример живьём.** Целое число — это «одна и больше цифр», удобно `цифра⁺`. «Имена, кроме ключевых слов» — `имя − ключевые_слова`. Дал имя `цифра = 0|1|…|9` — и дальше пишешь `цифра` вместо длинного перечисления.

💡 **По-человечески.** «Расширенные» = обычные регвыражения + пара удобных значков ради компактности. Новой теории нет.

# 13. Построение НКА по РРВ

🎓 **Чёткий ответ.** По заданному РРВ строят конечный автомат алгоритмом 4.1, работающим **рекурсивно по структуре выражения**: для элементарного символа — простейший автомат, а для составного автоматы подвыражений соединяются по правилам операций (объединение, конкатенация, итерация). Получается НКА с переходами по пустой цепочке `e`, которые затем устраняют. Вручную автомат собирают из готовых «кирпичиков».

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| НКА | недетерминированный КА (из точки может вести несколько дорог) |
| `e`-переход | «бесплатный прыжок» без чтения символа |
| рекурсивно | собираем большой автомат из маленьких кусочков |

🔬 **Пример живьём — кирпичики:**

| Кусок выражения | Что строим |
|---|---|
| символ `a` | один переход «прочитал `a` → принял» |
| `R₁\|R₂` | развилка: пускаем во вход обоих, сводим выходы |
| `R₁R₂` | хвост `R₁` соединяем со входом `R₂` |
| `R₁*` | петля назад + «бесплатный» пропуск (0 раз) |

💡 **По-человечески.** Шаблон превращаем в машинку: на каждую часть — кубик, кубики склеиваем как трубы.

# 14. Преобразование НКА в ДКА (детерминизация)

🎓 **Чёткий ответ.** Доказано: для любого НКА существует ДКА, распознающий тот же язык. Метод: состояниями ДКА становятся **группы (подмножества) состояний НКА**; старт — группа из начального состояния; для каждой группы и символа собирают группу, в которую переходит НКА. Заключительные группы — те, что содержат заключительное состояние НКА.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| НКА | автомат с развилками (несколько дорог из точки) |
| ДКА | автомат без развилок (ровно одна дорога) |
| группа состояний | «я могу быть сразу в нескольких комнатах» |

🔬 **Пример живьём.** НКА, ловящий слова, **кончающиеся на `1`** (состояния `A` — старт, `B` — приём; из `A` по `1` ведут два пути):

НКА:

| Сост. | `0` | `1` |
|---|---|---|
| →A | {A} | {A, B} |
| ⊛B | ∅ | ∅ |

Склеиваем в группы: старт `{A}`.

| Группа (= сост. ДКА) | `0` | `1` |
|---|---|---|
| →{A} | {A} | {A, B} |
| ⊛{A, B} | {A} | {A, B} |

Теперь из каждой клетки — **одна** дорога. Это ДКА.

💡 **По-человечески.** Было «шпионы разбегаются по развилкам» — стало «автобус везёт всю компанию как одно целое». Развилок нет, язык тот же.

# 15. Лексический анализ. Прямой и непрямой

🎓 **Чёткий ответ.** Лексический анализ (сканирование) — первый этап компиляции: символы группируются в **лексемы**; делает это **сканер**. Бывает **прямой** анализ (распознаём одну из всех лексем набором параллельных КА) и **непрямой**, синтаксически управляемый (процедуры вызываются по запросу синтаксического анализатора, который указывает ожидаемый тип). Непрямой экономнее по памяти.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| лексема | готовый смысловой кусочек: имя, число, ключевое слово |
| сканер | часть компилятора, делающая нарезку |

🔬 **Пример живьём.** Сканер режет поток `S:=(a+b)` на лексемы: `S`, `:=`, `(`, `a`, `+`, `b`, `)`.

| | Прямой | Непрямой |
|---|---|---|
| Кто командует | сканер сам | анализатор (старший) |
| Вопрос | «что это вообще?» | «это конкретно X?» |
| Память | копит таблицу всех лексем | таблицу не копит → экономнее |

Цепочка построения сканера: **регвыражение → НКА → ДКА → сканер**.

💡 **По-человечески.** Читая, ты видишь не буквы, а слова — сканер делает то же. Прямой перебирает все типы сам, непрямой проверяет только заказанный анализатором.

# 16. Перевод (трансляция) и модели простых трансляторов

🎓 **Чёткий ответ.** Перевод — соответствие $T \subseteq \Sigma^* \times \Delta^*$ (вход×выход); для ЯП должен быть **функцией** (один выход на вход). Простые модели: **конечный преобразователь** (КА + выход) и **МП-преобразователь** (МП-автомат со стопкой + выход).

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| перевод | машинка «слово вошло → другое слово вышло» |
| преобразователь | автомат с «принтером» (читает и печатает) |
| функция | один вход → ровно один выход |

🔬 **Пример живьём.**
- Простой перевод (конечный преобразователь): правило `0→a, 1→b`. Вход `010` → выход `aba`.
- Перевод посложнее (нужна стопка → МП-преобразователь): инфикс `a + b * c` → польская запись `a b c * +` (знак в конце, скобки не нужны).

💡 **По-человечески.** Перевод — переводчик слов. Преобразователь — автомат с принтером: простой (без памяти) или со стопкой (с памятью).

# 17. Синтаксический анализ. Разбор. Нисходящий и восходящий

🎓 **Чёткий ответ.** Синтаксический анализ строит структуру (дерево разбора). **Разбор** — последовательность применённых правил: **левый разбор** — правила в порядке левого вывода; **правый разбор** — обращение правил правого вывода. **Нисходящий** разбор строит дерево от корня к листьям (моделирует левый вывод, $R_l$); **восходящий** — от листьев к корню свёрткой (моделирует правый вывод, $R_r$).

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| разбор | список номеров применённых правил |
| нисходящий | сверху вниз, угадывая правила |
| восходящий | снизу вверх, сворачивая куски |

🔬 **Пример живьём.** Грамматика **Г2** (`(1) S→AB, (2) A→a, (3) B→b`), слово `ab`:
- **левый разбор** (раскрываем самый левый нетерминал): `S ⇒ AB ⇒ aB ⇒ ab` → правила `1, 2, 3`;
- **правый разбор** (раскрываем самый правый): `S ⇒ AB ⇒ Ab ⇒ ab` → правила `1, 3, 2`, записанные задом наперёд = `2, 3, 1`.

Один и тот же `ab`, но **левый разбор `1,2,3`, правый `2,3,1`**.

💡 **По-человечески.** Разбор — рецепт сборки слова в виде номеров правил. Нисходящий разворачивает «матрёшку» сверху, восходящий собирает снизу.

# 18. Левый и правый анализаторы

🎓 **Чёткий ответ.** Анализатор, реализующий нисходящий разбор, выдаёт левые разборы и называется **левым**; реализующий восходящий — выдаёт правые разборы и называется **правым**. Оба реализуются детерминированными МП-преобразователями: левый — модель **LL(k)**-анализатора, правый — модель **LR(k)**-анализатора (использует «перенос-свёртку»).

🔑 **Сложные слова.** ДМП-преобразователь — детерминированный МП-автомат со стопкой + выход, настроенный на разбор.

🔬 **Пример живьём.** Для `ab` (грамматика Г2):

| Анализатор | Идёт | Выдаёт | Модель |
|---|---|---|---|
| левый | сверху вниз | `1, 2, 3` | LL(k) |
| правый | снизу вверх | `2, 3, 1` | LR(k) |

💡 **По-человечески.** Просто названия двух стратегий: **LL — левый, сверху; LR — правый, снизу**.

# 19. Детерминированность. Свойства КС-языков и грамматик

🎓 **Чёткий ответ.** Детерминированный КС-язык распознаётся за один проход без возврата к прочитанному — такие языки эффективно транслируются. Свойства КС-грамматик: однозначность/неоднозначность, рекурсивность (лево- и праворекурсивность), бесполезные и недостижимые символы, приведённость, нормальные формы Хомского и Грейбах.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| детерминированный | читает слева направо, **не отматывает назад** |
| неоднозначность | у слова **два разных** дерева разбора |
| леворекурсия | правило `A → A…` (нетерминал слева зовёт себя) |
| бесполезный символ | из него нельзя вывести слово из одних букв |
| недостижимый символ | до него нельзя дойти от старта `S` |

🔬 **Пример живьём.**
- **Неоднозначность:** грамматика `E → E+E | E*E | a` для `a+a*a` даёт **два дерева** — `a+(a*a)` и `(a+a)*a`. Подставь `a=2`: `6` или `8`. Компилятор не знает, какой код делать → зло.
- **Бесполезный:** в `S → a | B`, `B → Bb` нетерминал `B` бесполезен (из него всегда остаётся `B`, до букв не дойти).
- **Недостижимый:** в `S → a`, `A → b` символ `A` недостижим (из `S` к нему дороги нет).

💡 **По-человечески.** Все эти свойства — проверка «годится ли грамматика для машины». Неоднозначность, левая рекурсия и мусор её портят.

# 20. Эквивалентные преобразования КС-грамматик

🎓 **Чёткий ответ.** Эквивалентные преобразования $G \to G'$ сохраняют язык ($L(G)=L(G')$) и убирают нежелательные свойства: устранение недостижимых символов, бесполезных, `e`-правил, цепных правил, левой рекурсии, факторизация, приведение к нормальным формам. Применяются, чтобы привести грамматику к виду для детерминированного анализатора.

🔑 **Сложные слова.** Цепное правило — вида `A→B` (один нетерминал прямо в другой). Факторизация — вынесение общего начала альтернатив.

🔬 **Пример живьём — устранение левой рекурсии (пошагово).** Грамматика `S → S a | b` (делает `b, ba, baa…`). Левая рекурсия `S→Sa` зацикливает нисходящий анализатор. Что делает: эта грамматика — «`b`, потом сколько-то `a`». Запишем это правой рекурсией с новым «хвостом» `T`:
```
Было:  S → S a | b
Стало: S → b T
       T → a T | e
```
Проверка — выводим `baa` по новой грамматике:
```
S → b T → b a T → b a a T → b a a   (T→e)
```
Слово то же → язык не изменился, но рекурсия стала правой, и нисходящий анализатор не зависает.

💡 **По-человечески.** «Причёски» для грамматики: меняем форму, язык оставляем. Левая рекурсия — главная боль, её перетасовывают в правую.

# 21. Лево- и правоанализируемые грамматики

🎓 **Чёткий ответ.** Грамматика **левоанализируема**, если для неё можно построить детерминированный **нисходящий** анализатор (выдаёт левые разборы). **Правоанализируема** — если можно построить детерминированный **восходящий** анализатор (правые разборы).

🔑 **Сложные слова.** Левый разбор — список правил вроде `1,2,3`; правый — вроде `2,3,1` (см. в.17).

🔬 **Пример живьём.** Для `ab` (Г2): можно построить и машину, выдающую левый разбор `1,2,3`, и машину, выдающую правый `2,3,1` → грамматика **и лево-, и правоанализируема**.

| Грамматика | Нисходящий | Восходящий | Итог |
|---|:---:|:---:|---|
| `S→AB, A→a, B→b` | ✅ `1,2,3` | ✅ `2,3,1` | обе |
| `S→Sa \| b` (левая рекурсия) | ❌ зацикливается | ✅ | только право |
| неоднозначная (2 разбора) | ❌ | ❌ | ни одной |

**Тонкость:** «только лево» невозможно (левоанализируемая всегда и правоанализируема, т.к. LL ⊂ LR).

💡 **По-человечески.** Вопрос «какой список (левый/правый) машина умеет выдавать без угадываний?». Левая рекурсия убивает нисходящий; неоднозначность — оба.

# 22. LL(k)-грамматика. Строгие и нестрогие

🎓 **Чёткий ответ.** Грамматика **LL(k)**, если по `k` символам впереди однозначно выбирается единственное правило раскрытия нетерминала. Признак: для правил `A→β`, `A→γ` должно быть `FIRSTₖ(βα) ∩ FIRSTₖ(γα) = ∅`. **Строгая (сильная) LL(k)** проверяет жёстче — через весь `FOLLOWₖ(A)`; класс уже, но анализаторы проще.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| аванцепочка | `k` символов, на которые подглядывают вперёд |
| FIRST(X) | с каких букв может **начинаться** `X` |
| FOLLOW(A) | какие буквы могут идти **сразу за** `A` |

🔬 **Пример живьём.** Грамматика **Г-пустая** `(1) S→aS`, `(2) S→e`:
- `FIRST(aS) = {a}`; правило 2 пустое → берём `FOLLOW(S) = {$}`.

Решение анализатора (1 символ вперёд):

| Вижу впереди | Беру правило |
|---|---|
| `a` | (1) `S→aS` |
| `$` (конец) | (2) `S→e` |

`{a} ∩ {$} = ∅` → выбор однозначен → **LL(1)**. Прогон `aa`: видим `a`→1, видим `a`→1, видим `$`→2 → разбор `1,1,2`.

💡 **По-человечески.** «Подсмотрел `k` букв → точно знаю правило». FIRST — «с чего начинается», FOLLOW — «что идёт следом». Не пересекаются → LL.

---

# 23. LR(k). Отношения Вирта–Вебера. Грамматики предшествования

🎓 **Чёткий ответ.** Грамматика **LR(k)** — если при восходящем разборе по разобранной части и `k` символам впереди однозначно находится **основа** (кусок для свёртки) и правило. Самый широкий класс. Отношения предшествования Вирта–Вебера `⋖ ≐ ⋗` размечают границы основы. На их основе — классы грамматик предшествования: простого, слабого, операторного.

🔑 **Сложные слова.**

| Значок | Смысл | Роль |
|---|---|---|
| `⋖` | «уступает» | **левый край** основы |
| `≐` | «равны» | **внутри** основы |
| `⋗` | «главнее» | **правый край** основы |
| основа | кусок, который пора свернуть | — |

🔬 **Пример живьём — скобки** `S → (S) | x`, слово `((x))`. Основу видно как самый «созревший» кусок:

| Строка | Основа | Правило | Стало |
|---|---|---|---|
| `( ( x ) )` | `x` | `S→x` | `((S))` |
| `( ( S ) )` | `(S)` | `S→(S)` | `(S)` |
| `( S )` | `(S)` | `S→(S)` | `S` |
| `S` | — | — | готово ✅ |

А отношения находят границы автоматически: расставляем `⋖ ≐ ⋗` между символами, ищем первый `⋗` (правый край), отступаем до `⋖` (левый край) — между ними основа.

**Классы предшествования:** простого (одно отношение между парой), слабого (шире), операторного (только между терминалами).

💡 **По-человечески.** Основа — «созревший» кусок для свёртки. Значки `⋖ ≐ ⋗` — разметка её краёв: нашёл `⋗`, отступил до `⋖` — вот основа.

---

# 24. Отношения между классами

🎓 **Чёткий ответ.** Между классами **собственное (строгое) включение**: внутренний класс целиком лежит во внешнем, но не совпадает. Каждая LL(k) есть LR(k); грамматики простого и слабого предшествования входят в LR(1); слабое и простое предшествование дают один класс языков; языки операторного предшествования — подкласс простого и не сравнимы с LL.

🔑 **Сложные слова.** Собственное включение — «целиком внутри, но не равно» (всегда есть элемент внешнего класса вне внутреннего).

🔬 **Пример живьём.**

```
┌─ КС-грамматики ───────────────────┐
│  ┌─ LR(k) ───────────────────────┐ │
│  │  ┌─ LL ─┐   ┌─ LR(1) ───────┐ │ │
│  │  │сверху│   │ ┌ предшеств. ┐ │ │ │
│  │  │ вниз │   │ └───────────┘ │ │ │
│  │  └──────┘   └───────────────┘ │ │
│  └───────────────────────────────┘ │
└─────────────────────────────────────┘
```

| Грамматика | LL? | LR? | Вывод |
|---|:---:|:---:|---|
| `S→aS \| e` | ✅ | ✅ | в пересечении |
| `S→Sa \| b` | ❌ | ✅ | **LR строго шире LL** |
| неоднозначная | ❌ | ❌ | вне обоих, но операторное берёт |

💡 **По-человечески.** Карта «кто в кого вложен». Грамматику переделывают (в.20) под более узкий класс — там анализатор проще.

# 25. Модель LL(k)-анализатора

🎓 **Чёткий ответ.** Модель LL(k)-анализатора — **детерминированный МП-преобразователь**, работающий по принципу **опустошения магазина**; эквивалентен **левому анализатору** ($R_l$). Состояние — **тройка (конфигурация)**: непрочитанный вход, содержимое магазина, выданный выход. Управляет **таблица** $M(X,u)$ (верх магазина + аванцепочка): замена, выброс, допуск, ошибка. Выход — левый разбор.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| тройка (конфигурация) | `(вход, магазин, выход)` — снимок состояния |
| такт (`⊢`) | один шаг работы |
| аванцепочка | символы впереди (при `k=1` — один) |

🔬 **Пример живьём — грамматика Г1** `(1) S→aS`, `(2) S→b`.

Управляющая таблица:

| Верх ↓ \ вход → | `a` | `b` | `$` |
|---|---|---|---|
| S | `aS`,1 | `b`,2 | ошибка |
| a | выброс | ошибка | ошибка |
| b | ошибка | выброс | ошибка |
| $ | ошибка | ошибка | допуск |

(`aS,1` = заменить `S` на `aS` и выдать `1`.)

Потактный прогон `aab` (тройка `(вход, магазин, выход)`):
```
(aab, S$,  e)
 ⊢ (aab, aS$, 1)    ← (S,a)=aS,1: замена, выдали 1
 ⊢ (ab,  S$,  1)    ← (a,a): выброс
 ⊢ (ab,  aS$, 11)   ← (S,a)=aS,1: замена
 ⊢ (b,   S$,  11)   ← выброс a
 ⊢ (b,   b$,  112)  ← (S,b)=b,2: замена
 ⊢ (e,   $,   112)  ← выброс b
 ⊢ ДОПУСК ✅           ← ($,$): допуск
```
Выход `112` — левый разбор `aab` (правила 1,1,2). Проверка: `S⇒aS⇒aaS⇒aab`. ✅

💡 **По-человечески.** Смотрит на верх стопки + символ впереди → лезет в таблицу → делает 1 из 4 ходов. Цель — опустошить магазин до `$`.

# 26. Построение управляющих таблиц LL(k)-анализаторов

# 27. Модель LR(k)-анализатора

🎓 **Чёткий ответ.** Модель LR(k)-анализатора — **детерминированный МП-преобразователь** (эквивалентен **правому анализатору**, $R_r$). Верх магазина — справа, за такт цепочка `α` вверху заменяется на `β`. Управляет таблица из LR(k)-таблиц с двумя функциями: **f** (действие: перенос / свёртка `i` / допуск / ошибка) и **g** (переход — имя следующей таблицы). Читает слева направо, выдаёт правый разбор.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| перенос | закинуть символ в магазин |
| свёртка | собрать верхний кусок (основу) в нетерминал |
| f / g | функция действия / функция перехода |

🔬 **Пример живьём — грамматика Г2** `(1) S→AB, (2) A→a, (3) B→b`, слово `ab` (верх магазина справа):

| Магазин | Вход | Действие |
|---|---|---|
| | `ab$` | перенос `a` |
| `a` | `b$` | свёртка `A→a` (2) |
| `A` | `b$` | перенос `b` |
| `A b` | `$` | свёртка `B→b` (3) |
| `A B` | `$` | свёртка `S→AB` (1) |
| `S` | `$` | **допуск** ✅ |

Свёртки `2, 3, 1` — это **правый разбор** `ab` (см. в.17).

💡 **По-человечески.** Зеркало LL: строит дерево снизу, складывая буквы в нетерминалы. Две операции — перенос (закинуть) и свёртка (собрать).

# 28. Алгоритмы типа «перенос-свёртка»

🎓 **Чёткий ответ.** Алгоритм «перенос-свёртка» подобен LR(k)-анализатору, но вместо LR(k)-таблиц использует **матрицы отношений предшествования**, а вместо функций действия/перехода — функции **переноса** и **свёртки**. Формально пара $A=(f,g)$: `f` решает «перенести или свернуть», `g` — каким правилом свернуть. Символы переносят в магазин, пока вверху не образуется основа, затем сворачивают.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| f — функция переноса | «нести символ или сворачивать?» |
| g — функция свёртки | «если сворачивать — то каким правилом?» |

🔬 **Пример живьём.** Скобки `S→(S)|x`, слово `(x)`:

| Магазин | Вход | f-решение |
|---|---|---|
| | `(x)$` | перенос `(` |
| `(` | `x)$` | перенос `x` |
| `( x` | `)$` | свёртка `x→S` (основа `x`) |
| `( S` | `)$` | перенос `)` |
| `( S )` | `$` | свёртка `(S)→S` (основа `(S)`) |
| `S` | `$` | допуск ✅ |

💡 **По-человечески.** Тот же восходящий разбор, но «облегчённый»: решения — по простой матрице, а не по большим таблицам. `f` = «нести/сворачивать», `g` = «чем сворачивать».

# 29. Построение «перенос-свёртка» для грамматик предшествования

🎓 **Чёткий ответ.** Алгоритмы строят для трёх классов: 6.6 — простого предшествования, 6.7 — слабого, 6.8 — операторного. Матрицу отношений `⋖ ≐ ⋗` формируют заранее. Функция переноса определяется прямо по матрице, функция свёртки — по верхней части магазина.

🔑 **Сложные слова.** Матрица предшествования — таблица, какое отношение (`⋖`/`≐`/`⋗`) между парой символов.

🔬 **Пример живьём — правило для функции f:**

| Отношение «верх магазина ↔ вход» | Действие f |
|---|---|
| `X ⋖ a` (левый край ещё впереди) | **перенос** |
| `X ≐ a` (внутри основы) | **перенос** |
| `X ⋗ a` (правый край — пора!) | **свёртка** |

То есть: пока не встретили `⋗` — переносим; встретили `⋗` — сворачиваем. `g` смотрит на верх магазина и находит подходящее правило.

| Алгоритм | Класс |
|---|---|
| 6.6 | простого предшествования |
| 6.7 | слабого |
| 6.8 | операторного |

💡 **По-человечески.** Значки `⋖ ≐ ⋗` превращаются в команды: дошёл до `⋗` → нашёл правый край основы → сворачивай.

# 30. СУ-трансляция для генерации кода и интерпретации

🎓 **Чёткий ответ.** Синтаксически управляемая трансляция реализует перевод прямо при разборе. Формы: **СУ-определения (СУО)** — грамматика + атрибуты + семантические правила (аннотированное дерево); **схемы трансляции (СУТ)** — семантические действия в теле правил. Атрибуты: **синтезируемые** (снизу вверх) и **наследуемые** (сверху/слева). Результат — **промежуточная программа** (польская запись, триады, тетрады), которую интерпретируют или переводят в ассемблерный код.

🔑 **Сложные слова.**

| Слово | Значение |
|---|---|
| атрибут | пометка на узле дерева (тип, значение, адрес) |
| синтезируемый | считается снизу (от потомков) |
| наследуемый | приходит сверху/слева |
| промежуточная программа | дерево, переписанное в линейный вид |

🔬 **Пример живьём — выражение `a + a * a` в разных формах:**

| Форма | Запись |
|---|---|
| польская (постфикс) | `a a a * +` |
| триады | (1) `* a a`, (2) `+ a (1)` |
| тетрады | `T1 := a * a`, `T2 := a + T1` |

Все три кодируют одно дерево. Дальше: либо **интерпретация** (сразу выполнить), либо перевод в **ассемблерный код**.

💡 **По-человечески.** Разбираем дерево — и тут же по привязанным действиям генерируем код. Промежуточная программа — «полуфабрикат», из которого легко сделать и интерпретацию, и ассемблер.
